## 專案背景

為銀行客戶開發的資產總覽系統，整合銀行各類金融資產（存款、基金、股票、保險等），以統一介面呈現給用戶。系統需承受高併發查詢，對可用性與效能要求極高。

## 我的角色

擔任後端工程師，參與系統架構設計與核心功能實作。

## 架構設計

採用 DDD（領域驅動設計）拆分業務邊界：
- 各資產類型（存款、基金、股票）各自為獨立 Bounded Context
- 使用 Aggregate 封裝業務邏輯，避免貧血模型
- Application Service 協調跨 Context 的資料組合

## 快取策略

為應對高併發查詢，設計雙層快取架構：

1. **本地快取（Caffeine）**：極低延遲，用於高頻、短時間內重複的查詢
2. **分散式快取（Redis Cluster）**：跨節點共享，確保水平擴展後的快取一致性

快取失效策略與更新時機依各資產類型的資料特性分別設計。

## 測試

- 使用 JMeter 建立壓力測試情境，模擬高併發 API 請求
- 進行黑箱測試（功能驗收）與白箱測試（程式碼邏輯驗證）
- 根據測試結果反覆調整快取參數與 SQL 查詢效能

## 心得

這是規模最大、技術複雜度最高的專案。DDD 的學習曲線不低，但一旦掌握後，業務邏輯的表達變得非常清晰。快取設計讓我深刻理解「快取不是萬靈丹」——過期策略、資料一致性、記憶體壓力，每一個面向都需要仔細權衡。
